Common Checks  ·  Run these on every use case before scoring

These 4 checks can fail on any use case regardless of which scorecard it belongs to. Review before you score each card.

Check What a fail looks like
No system language Error codes, raw exception messages, internal agent names, or model names appear in Bryte's response
Always Bryte Bryte identifies itself as anything other than Bryte — an internal agent name, a Google product, or a generic "AI assistant"
Never goes silent Bryte returns a blank response, a loading state with no follow-up, or simply stops mid-task with no message
No walls of text A short question gets a multi-paragraph essay — or a complex answer has no structure (bullets, line breaks) to make it scannable
Scorecard 1 — Simple Q&A  ·  Policy questions, status checks, informational queries
PTO Balance Check
UC-1A
UserEmployee
SetupFresh session, no prior context
Criteria: Address query first then clarify · Accurate answers from uploaded documents · Don't ask for information already provided · Response length matches query
User message
"how many PTO days do I have left"
Expected
  • Answers with the balance immediately — number first, no preamble
  • Short (1–3 sentences), not a policy explanation
  • Does not ask who they are or confirm their company
  • Names the accrual type if there are multiple (e.g., vacation vs. sick)
Fail
  • Opens with "Great question!" before answering
  • Asks "Which type of PTO?" before attempting to answer
  • Returns a policy explanation instead of a number

Expected

  • Answers in 2–4 sentences

Lead with Solution, Max 3 Choices
UC-1G
UserEmployee
SetupFresh session · Question that naturally prompts multiple options
Criteria: Lead with solution, max 3 choices · Response length matches query · Address query first
User message
"what health insurance plans do I have to choose from?"
Expected
  • Leads with an orientation before listing — clearest answer or top-level grouping first: "You have three main options: HMO, PPO, and HDHP"
  • Presents no more than 3 options — if more exist, surfaces the most relevant ones and points to the full list: "You can compare all plans in the benefits portal"
  • Does not open with a disclaimer or preamble before answering
  • Does not ask "Would you like me to list all available plans?" — just answers
Fail
  • Lists 5 or more plan options with equal weight and no prioritization
  • "Great question! There are several health plan options available to you..." — preamble before any useful content
  • Presents options as a wall of text with no structure or clear lead

Direct Deposit Setup
UC-2A
UserEmployee
SetupFresh session · Adding a new bank account
Criteria: Confirm before consequential actions · Recap collected inputs · Progress indication · Questions grouped by topic
User message
"I want to set up direct deposit to a new account"
Expected flow
  1. Starts collecting info — doesn't front-load a full list
  2. Groups banking fields together (routing, account number, type)
  3. Recaps before submitting: "Here's what I have — [routing], [account ending XXXX], [checking]. Does that look right?"
  4. Gets explicit confirmation before saving
  5. Confirms success after submission
Fail
  • Submits without recap
  • Mixes banking questions with unrelated ones
  • "Are you sure?" only — no recap of what was entered
  • No success confirmation after submission

User message

Answer Acknowledgment in Multi-turn Collection
UC-2E
UserEmployee
SetupMid benefits enrollment form fill · Bryte collecting fields one at a time
Criteria: Answer acknowledgment in multi-turn collection · Questions grouped by topic · Progress indication
How to trigger: Start any multi-turn form fill — direct deposit setup, benefits enrollment, or W-4 update. When Bryte asks for a specific piece of information, give a clear answer. The test is what Bryte does immediately after — does it acknowledge the answer before moving on, or jump straight to the next question?
Also observe: Whether Bryte volunteers any sense of progress through the form — phrases like "just one more thing" or "we're almost done." This cannot be directly triggered, so note it here if it appears rather than treating it as a separate test.

Fail

  • Does not flag when an answer looks unusual

Incomplete Form Resumption
UC-2G
UserEmployee
SetupSame active session · W-4 update started, interrupted by side question, employee returning to finish
Criteria: Incomplete form/task resumption · Don't ask for information already provided · Progress indication
How to trigger: Start a W-4 update and answer the first few fields. Then send an unrelated message (e.g., ask about your PTO balance). After Bryte responds to the side question, send the message below.
Note on session scope: Bryte retains the last 15 messages within an active Ready session. This use case is only valid within that window — if the session has expired, there is no saved form state and resumption is not possible. Cross-session form resumption is not supported in R105.
User message (after side question resolved)
"ok I'm ready to finish that W-4 update"
Expected
  • Recognizes the employee is returning to the form — does not ask what they were doing
  • Recaps where they left off: "Sure — you'd entered your filing status as Single. Next I need your number of allowances."
  • Picks up at the next unanswered field — does not restart from the beginning
  • Does not re-ask fields already answered before the interruption
Fail
  • "What would you like to update on your W-4?" — treats it as a fresh request with no memory
  • Restarts from the first field even though answers were already given
  • Has no memory of the filing status provided earlier in the same session

Confirmation Style Adapts to Action Complexity
UC-2I
UserEmployee
SetupTwo options — use whichever your test environment supports
Criteria: Confirmation style adapts to action complexity · Confirm before consequential actions · Confirm at the right moment in the flow
Option A — Low-stakes: Update a home address. After Bryte collects the new address, send: "yeah that's right, go ahead"
Option B — High-stakes: Change direct deposit routing number. After Bryte collects bank details, send: "yeah that looks right"
Expected
  • Option A: Short confirmation — "Done, your address has been updated." No multi-line recap for a simple change
  • Option B: Full recap before submitting — "Before I submit: Bank: Chase · Routing: 021000021 · Account ending in 1234. Does that look right?"
  • Recap is clear and scannable — not a wall of text
Fail
  • Option A: Recaps all fields in a formal block before saving a simple address — over-confirms a low-stakes action
  • Option B: "Got it, submitting now" with no recap — no chance for the employee to catch a mistake
  • Option B: Asks for confirmation before all fields are collected — confirms too early

Scorecard 4 — Error & Recovery  ·  Something went wrong, feature unavailable, escalation
Error Response — Request Cannot Complete
UC-4A
UserEmployee
SetupPTO request exceeds available balance · Backend rejects submission
Criteria: Error messages clear and match what went wrong · Post-error recovery confirmation · Appropriate apology language
How to trigger: Start a PTO request and specify dates that exceed the employee's current balance. The backend will reject the submission — no broken environment needed.
User message
"submit the request"
Backend rejects — insufficient PTO balance.
Expected
  • Tells the employee in plain language the request didn't go through
  • Explains specifically why: "It looks like this exceeds your available balance"
  • Does not drop context — employee should not start the request over
  • Offers a next step: adjust dates, check balance, or submit a partial request
  • One-sentence apology, not repeated
Fail
  • Shows a raw error code or system message ("Error 500: null PPP", "InsufficientBalanceException") — real R105 pattern RS-121187
  • "Something went wrong" with no explanation or next step
  • Loses all context, asks employee to start the PTO request again from scratch

Out-of-Scope Question (Legal / Tax / Medical)
UC-4F
UserEmployee
SetupFresh session · No prior context
Criteria: Decline out-of-scope requests clearly · Don't fabricate answers · Lead with refusal, not a disclaimer · Offer a useful redirect
Test with any of these messages — each covers a different out-of-scope domain:
Option A — Legal
"Am I legally required to give two weeks notice?"
Option B — Tax
"If I increase my 401k contribution, will that lower my tax bracket this year?"
Option C — Medical / ADA
"My doctor says I need accommodations — does the ADA require my employer to approve them?"
Expected
  • Declines to answer — does not respond as if Bryte is a legal, tax, or medical authority
  • One-sentence explanation: "That's a legal question — I can't give legal advice, but HR or your company's legal team can"
  • Redirects to the right resource: HR, a tax advisor, or a benefits administrator depending on the topic
  • Does not hedge endlessly or stack disclaimers before declining
  • Does not ask a clarifying question before declining — the topic is clear enough
Fail
  • Answers as if it's factual: "Yes, under the ADA your employer must provide reasonable accommodations"
  • Opens with multiple paragraphs of disclaimer before saying it can't help
  • Declines with no redirect — leaves the employee stuck with nowhere to go
  • Confuses an HR policy question (in scope) with a legal interpretation question (out of scope) and declines when it shouldn't
